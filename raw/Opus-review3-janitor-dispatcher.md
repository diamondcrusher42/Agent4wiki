# comment on janitor.md

This is the Janitor's core logic — the gate between clone output and merge. Let me review it against the architecture and the other three files.
What's right:
The three-tier directive system (BLOCK → SUGGEST → NOTE) maps cleanly to the red/yellow/green severity framework from hstack. The circuit breaker (3 retries then escalate to human) prevents infinite loops — a clone that fails three times isn't going to succeed on the fourth try with the same template. The WikiScythe as a separate class from the Auditor is correct separation of concerns: code review is synchronous (happens after every clone), wiki maintenance is asynchronous (runs on schedule).
The closing loop description at the bottom is the clearest statement yet of how the pieces connect: Clone finishes → Keychain sweeps → Janitor audits → merge or block → Forge logs. That's the production pipeline.
Issues to address:
The detectsArchitecturalMess method is too naive for production. Keyword matching on "hacky" and "todo" will miss real architectural problems and false-positive on legitimate TODOs. The comment says "this might be a fast BitNet/Haiku LLM check" — that's the right answer. In the MVP, a more robust approach: parse the clone's janitor_notes and files_modified list, check if the clone touched files outside its stated objective (scope creep), check if it created files without tests, check if it modified shared config files. These are structural checks, not semantic ones, and they don't need an LLM.
The Janitor needs the two handshake fields I flagged in the TASK template review: tokens_consumed and duration_seconds. Without them, the Janitor can say "this passed" but can't say "this passed but took 3x longer than the last similar task." The Forge needs this data, and the Janitor is the one who receives it in the handshake. The Janitor should log these to the event stream even on NOTE directives.
Missing: the connection to the dispatcher. The dispatcher's execute_task function runs the clone and gets a result, but it doesn't call the Janitor. The lifecycle should be:
dispatcher.execute_task()
  → clone runs, produces handshake JSON
  → dispatcher extracts handshake from clone stdout
  → janitor.evaluateMission(handshake, retryCount)
  → if BLOCK: move to failed, optionally re-queue with retry count + 1
  → if SUGGEST: re-queue with janitor feedback injected into the task
  → if NOTE: merge worktree, log to wiki
Right now the dispatcher treats every result as either COMPLETED or FAILED. It needs to parse the clone's handshake JSON from stdout, feed it to the Janitor, and act on the directive. This is the integration point between dispatcher.py and this auditor.ts.
The TypeScript vs Python tension surfaces again here. The dispatcher is Python. The Janitor is TypeScript. In production, the dispatcher would need to call the Janitor either via subprocess (Node.js script), via MCP, or by porting the Janitor to Python. The cleanest path: port the Janitor to Python since it's a relatively small amount of logic (evaluate handshake, check patterns, return directive). The WikiScythe already needs the MemoryStore, which you'd also want in Python for MemPalace compatibility.
The WikiScythe is a skeleton. The pruneStaleKnowledge method has comments but no logic. The three operations it needs are: find entries where valid_until has passed and delete them, search for contradictory tags (this requires semantic comparison, not just string matching — MemPalace's contradiction detection handles this), and find orphan wiki pages with no inbound wikilinks (a simple regex scan of all [[page-name]] references across the wiki, then find pages that nobody links to). These are implementable now with the MemoryStore interface.
Missing: the SUGGEST retry loop. When the Janitor issues SUGGEST, what happens? The clone needs to be re-launched with the Janitor's feedback appended to the mission brief. The dispatcher needs to handle this: take the original task, append janitor_feedback to the objective, increment a retry counter, and drop it back in the inbox. This creates the feedback loop: clone tries → Janitor suggests → clone retries with guidance → Janitor re-evaluates.
The Forge logging at the end is exactly right but it should be structured, not just appending a string to wiki/log.md. The Forge needs machine-parseable data. Write the NOTE to forge/errors/database.json (or a JSONL file) with: task_id, skill, directive, notes, tokens_consumed, duration, files_modified. This is what the Forge pattern-matches against to find improvement opportunities.
